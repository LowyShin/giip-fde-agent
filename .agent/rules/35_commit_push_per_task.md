# 작업(태스크)마다 커밋·Push 필수 (MANDATORY)

> **HARD RULE** — 이 규칙은 기본 동작보다 우선한다.

## 핵심 규칙
하나의 작업(태스크)을 끝낼 때마다 **반드시 커밋하고 push**한다.
목적은 **언제든 그 시점으로 되돌릴(restore) 수 있도록** 이력을 남기는 것이다.

- 작업 단위 = 태스크 1건 또는 의미 있는 변경 1건.
- 여러 파일을 바꿨더라도 **작업이 끝나면 그 작업 전체를 하나의 커밋**으로 묶는다.
- 커밋만 하고 push를 미루지 않는다. 커밋 → push까지가 1작업의 마무리다.

## 절차
1. 작업 완료 후 변경 파일을 스테이징하고 커밋한다.
   - 커밋 메시지는 [11_structured_commit.md](11_structured_commit.md)를 따르고, 태스크 번호가 있으면 포함한다.
     예: `task(<id>): <요약>`
2. 커밋 직후 push한다.
   - 프로젝트에 **인증 래퍼 스크립트가 정의돼 있으면 그것을 사용**한다(직접 `git push` 대신).
   - 없으면 저장소에 설정된 정상 push 방법(SSH/PAT 등)으로 `git push`한다.
   - 기본 브랜치 히스토리를 재작성하는 force push는 명시적 지시 없이는 금지.
3. push 성공 확인 후 완료 보고한다(파일 수정 시 [33_repo_url_in_report.md](33_repo_url_in_report.md)에 따라 원격 URL 포함).

## Restore(되돌리기) 기준
- 각 작업이 독립 커밋으로 남으므로, 문제가 생기면 해당 커밋 해시로 되돌린다.
  - 특정 파일만: `git checkout <commit> -- <path>`
  - 전체 이력 확인: `git log --oneline`으로 작업별 커밋 확인 후 대상 선택.

## 금지 사항
- ❌ 여러 작업을 몰아서 한 번에 커밋/push (작업별 되돌리기가 불가능해짐)
- ❌ 커밋만 하고 push 생략
- ❌ 작업 완료 후 커밋·push 없이 다음 작업으로 진행

**Why:** 작업이 독립 커밋으로 남아야 문제 발생 시 그 시점으로 정확히 되돌릴 수 있다. 여러 작업을 몰아서 커밋하면 restore 단위가 사라진다.
